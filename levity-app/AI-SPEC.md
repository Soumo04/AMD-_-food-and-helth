## 1. System Classification

**System Type:** Autonomous Agent / Stateful Workflow

**Description:**
Levity's "Zero-G Engine" is a predictive, context-aware autonomous agent. It ingests real-time biometric data (Oura/Apple Health via Firebase Auth), calendar schedules, and geolocation to predict nutritional vulnerabilities (e.g., cortisol spikes, empty pantries). Using the "GSD Protocol", it acts on these predictions by interacting with external APIs (Instacart, Sweetgreen) via Firebase Cloud Functions. "Good" looks like accurate predictions of user needs with zero-friction, correct logistical execution.

**Critical Failure Modes:**
1. **Financial Execution Error:** Executing an unauthorized or incorrect food order via external APIs.
2. **Context Hallucination:** Incorrectly synthesizing biometric and schedule data, leading to inappropriate nudges (e.g., suggesting a high-carb meal to a diabetic user).
3. **Privacy Violation:** Leaking sensitive health data across boundaries or to unauthorized third-party services.

## 1b. Domain Context

**Industry Vertical:** Health, Wellness, and Nutrition Logistics
**User Population:** High-stress professionals, athletes, and individuals experiencing decision fatigue around daily nutrition.
**Stakes Level:** High (Involves PII, sensitive health data, and automated financial transactions).
**Output Consequence:** If the AI errs, the user could be charged for unwanted food, receive meals that violate dietary restrictions, or be given inappropriate health nudges.

### What Domain Experts Evaluate Against

| Dimension | Good (Expert Accepts) | Bad (Expert Flags) | Stakes | Source |
|-----------|-----------------------|--------------------|--------|--------|
| **Predictive Accuracy** | Accurately identifies stress/caloric deficit patterns from combined inputs. | Recommends a high-sugar snack during a low-activity rest period. | High | Nutritionists |
| **Logistics Safety** | Orders fall within preset user budget and dietary restrictions. | Orders an allergen-containing meal or exceeds the $50 budget limit. | Critical | E-commerce Risk |
| **Intervention Timing** | Nudges arrive exactly when vulnerability is highest (e.g., leaving the gym). | Sends food ordering nudges at 3 AM or during an active meeting. | Medium | Behavioral Psychology |

### Known Failure Modes in This Domain
- **The "Over-Automation" Trap:** Ordering too much perishables that go bad before the user can eat them.
- **Biometric Misinterpretation:** Misinterpreting a high heart rate from an exciting movie as a stress/cortisol event requiring a dietary intervention.

### Regulatory / Compliance Context
- **HIPAA / GDPR:** Health data must be handled securely; biometric streams cannot be used for unauthorized profiling.
- **Financial Compliance:** Automated API orders require explicit opt-in and strict spending limits.

### Domain Expert Roles for Evaluation

|------|---------------|
| Clinical Dietitian | Reviewing the mapping logic between biometric signals and nutritional interventions. |
| QA Engineer (Financial) | Auditing the Human-in-the-Loop (HITL) checkpoints for the GSD Protocol ordering functions. |

## 2. Framework Decision

**Selected Framework:** LangGraph
**Version:** v0.2.x

**Rationale:**
Levity requires long-running, fault-tolerant state management and Human-in-the-Loop (HITL) approvals. LangGraph's graph-based architecture allows us to model the user's day as a continuous state machine. Crucially, its native checkpointing enables the system to pause execution (e.g., "Wait for user to approve the $64.20 Instacart order") and resume reliably. LangGraph also seamlessly supports Firebase MCP servers as tool nodes.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| the agent Agent SDK | Best for pure MCP and coding tasks, but lacks the robust, persistent time-travel checkpointing needed for day-long health monitoring workflows. |
| OpenAI Agents SDK | Locks us into OpenAI; we need flexibility to use Gemini (for deep Google ecosystem integration) or Claude (for complex reasoning). |
| CrewAI | Checkpointing is too coarse; difficult to implement strict state transitions and financial guardrails. |

**Vendor Lock-In Accepted:** No. LangGraph allows us to swap underlying LLMs (Gemini, Claude, GPT-4o) depending on cost and reasoning requirements.

## 3. Framework Quick Reference

### Installation
```bash
pip install langgraph langchain-core langchain-openai firebase-admin mcp
```

### Core Imports
```python
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator
```

### Entry Point Pattern
```python
class UserState(TypedDict):
    biometrics: dict
    pantry_status: dict
    pending_order: dict
    messages: Annotated[list, operator.add]

def predict_needs(state: UserState):
    # Call LLM with Firebase MCP context
    pass

def execute_order(state: UserState):
    # Trigger Firebase Cloud Function
    pass

def requires_approval(state: UserState):
    if state.get("pending_order"): return "wait_for_user"
    return END

graph = StateGraph(UserState)
graph.add_node("predict", predict_needs)
graph.add_node("order", execute_order)
graph.set_entry_point("predict")
graph.add_conditional_edges("predict", requires_approval)
app = graph.compile(checkpointer=memory)
```

### Common Pitfalls
1. **State Overwrite:** Forgetting to use `Annotated[list, operator.add]` for lists, causing previous context to be overwritten.
2. **Missing Checkpointers:** Running the graph without a checkpointer means long-running health monitoring states will be lost if the server restarts.

## 4. Implementation Guidance

**Model Configuration:**
- **Primary Model:** Gemini 1.5 Pro (for complex context window synthesis of daily logs).
- **Temperature:** 0.1 (We need highly deterministic predictions based on health data, not creative storytelling).

**Core Pattern:**
State machine with continuous background evaluation and explicit interrupt nodes for financial transactions.

**Tool Use:**
Firebase MCP will expose tools like `read_biometrics`, `check_pantry`, and `trigger_instacart`. The agent will invoke these tools within specific nodes.

## 4b. AI Systems Best Practices

### Structured Outputs with Pydantic
Always enforce structured outputs when determining dietary interventions to ensure downstream systems (like Instacart APIs) receive exact, parseable parameters.

```python
from pydantic import BaseModel, Field

class DietaryIntervention(BaseModel):
    action_type: str = Field(description="e.g., 'snack_nudge', 'grocery_order'")
    recommended_items: list[str] = Field(description="List of food items")
    urgency_level: str = Field(description="low, medium, high")
    estimated_cost: float = Field(description="Cost in USD")
```

### Async-First Design
Biometric streams and MCP tool calls are I/O bound. Ensure all LangGraph nodes use async implementations (`ainvoke`) to prevent blocking the main server thread.

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|---------------------|----------|
| **Financial Safety** | Pass/Fail: Did it respect budget? | Code Metric | Critical |
| **Context Faithfulness**| 1-5: Does recommendation match biometrics? | LLM Judge | High |
| **Tool Execution** | Pass/Fail: Did it call the correct MCP tool? | Code Metric | Critical |
| **Tone/Style** | 1-5: Is the nudge calming and "Zero-G"? | LLM Judge | Medium |

### Eval Tooling

**Primary Tool:** LangSmith + Braintrust (for rigorous dataset management).

**Setup:**
```bash
export LANGCHAIN_TRACING_V2=true
export LANGCHAIN_PROJECT="levity-zero-g-engine"
```

### Reference Dataset
**Size:** 25 curated "Day in the Life" user scenarios.
**Composition:** 
- 10 baseline days (normal stress, adequate food).
- 10 vulnerability scenarios (high cortisol, empty pantry).
- 5 edge cases (post-marathon recovery, sick day).
**Labeling:** Domain experts (Dietitians) provide the "Golden Responses" for what the intervention *should* be.

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Budget Limit** | `estimated_cost > $50` | Block & Escalate to User |
| **Allergy Check** | `item in user_allergies` | Block & Regenerate |
| **Frequency Cap** | `nudges_today > 3` | Suppress Notification |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **Dismissal Rate**| All nudges dismissed by user | Flag scenario for prompt tuning |
| **Order Rejections**| Orders cancelled by user post-prompt | Review LLM context synthesis |

## 7. Production Monitoring

**Tracing Tool:** LangSmith (native LangGraph integration).

**Key Metrics to Track:**
1. End-to-end latency of the prediction node.
2. Token consumption per user per day.
3. MCP Tool call success rate.

**Smart Sampling Strategy:**
Automatically flag and send for human review any trace where an intervention was recommended but the user immediately dismissed the notification.
